# 《闺蜜》第一阶段开发内容清单

## 1. 第一阶段目标

第一阶段只验证一个核心问题：

> 用户是否愿意上传照片，并持续为了“自己穿上后的 AI 试穿效果图”打开闺蜜。

第一阶段必须形成完整闭环：

```text
微信登录
→ 建立基础资料
→ 上传三视图
→ 身材分析
→ 选择穿搭场景
→ 生成今日试穿效果图
→ 用户评分反馈
→ 沉淀用户偏好
```

## 2. 第一阶段不做什么

为了避免 MVP 失焦，第一阶段暂不做：

- 社区广场。
- 穿搭 PK。
- 完整电商商城。
- 复杂虚拟形象系统。
- 复杂 AI 聊天闺蜜。
- 完整衣橱运营系统。
- 大规模趋势库后台。

第一阶段可以预留入口和数据结构，但不做重功能。

## 3. 用户端页面

### 3.1 首页今日推荐

当前优先级：P0

功能：

- 展示日期、天气、幸运色。
- 展示用户今日 AI 试穿效果图。
- 展示今日穿搭标题。
- 展示图上关键理由标注。
- 支持主题切换：米色、粉色、酷炫。
- 支持场景切换：上班、约会、聚会、出游、更多。
- 展示生成额度：免费 2 张 / 付费 10 张。

验收标准：

- 首屏核心必须是试穿图，不是文字说明。
- 场景控件不能遮挡人物头部。
- Logo 作为水印，不能强遮挡内容。
- 切换主题后人物图和信息仍清晰。

### 3.2 三视图采集页

当前优先级：P0

功能：

- 上传正面照。
- 上传侧面照。
- 上传背面照。
- 展示拍摄要求。
- 上传前展示隐私授权提示。
- 支持重新拍摄 / 重新上传。

验收标准：

- 三张图未齐时不能进入分析。
- 用户知道照片用途：身材分析、试穿生成、衣橱识别。
- 用户知道照片可删除。

### 3.3 身材分析结果页

当前优先级：P0

功能：

- 展示身材比例分析结果。
- 展示适合的穿搭策略。
- 展示应避免的版型风险，但不使用羞辱性表达。
- 生成用户 `Body_Profile`。

验收标准：

- 不出现“胖、矮、腿短、缺陷”等表达。
- 每个判断都给出穿搭策略。
- 数据可被推荐系统使用。

### 3.4 生成中页面

当前优先级：P0

功能：

- 展示“米粒正在帮你搭今天这套”之类的生成状态。
- 显示当前生成步骤：分析场景、筛选单品、生成试穿图。
- 处理生成失败。

验收标准：

- 生成失败时能重试。
- 生成失败不扣额度，除非后端明确已经生成成功。

### 3.5 今日结果页 / 试穿详情页

当前优先级：P0

功能：

- 展示高清试穿图。
- 展示推荐理由：天气、比例、颜色、场景。
- 展示使用单品来源：用户衣橱 / 趋势库 / 商品候选库。
- 支持评分。
- 支持收藏。
- 支持重新生成。

验收标准：

- 用户能清楚知道为什么这样穿。
- 用户没有主动点击时，不展示购买链接。
- 如用户点击“找类似款”，同类型衣服必须展示多个候选链接。

### 3.6 评分反馈页

当前优先级：P0

功能：

用户对生成结果评分：

- 喜欢程度。
- 是否适合自己。
- 是否愿意现实穿。
- 是否时尚。
- 图片像不像自己。

验收标准：

- 评分写入用户偏好画像。
- 低分必须触发原因采集。

### 3.7 我的基础资料页

当前优先级：P1

功能：

- 昵称、头像、城市。
- 身高、体重。
- 星座。
- 偏好颜色。
- 不喜欢的颜色。
- 风格关键词。
- 会员状态。
- 今日剩余生成次数。

验收标准：

- 用户可修改基础资料。
- 资料变化会影响下一次推荐。

## 4. 第一阶段服务模块

### 4.1 用户服务

当前优先级：P0

能力：

- 微信登录。
- 创建 `User_ID`。
- 保存基础资料。
- 查询会员状态。
- 查询每日生成额度。

### 4.2 上传与文件服务

当前优先级：P0

能力：

- 上传三视图。
- 上传自拍或穿搭图。
- 图片私有存储。
- 生成短期访问 URL。
- 删除用户图片。

### 4.3 身材分析服务

当前优先级：P0

第一阶段允许先用 Mock 或半自动结果，接口必须按真实结构设计。

输出：

- 肩部比例。
- 腰臀比例。
- 腿长比例。
- 头身比。
- 身材标签。
- 穿搭策略标签。
- 风险标签。

### 4.4 推荐编排服务

当前优先级：P0

推荐顺序：

```text
用户场景
→ 天气
→ 身材策略
→ 用户衣橱
→ 趋势库
→ 商品候选库
→ 生成推荐方案
```

规则：

- 有衣橱先用衣橱。
- 衣橱不满足时用趋势库补。
- 没有衣橱时直接用趋势库。
- 不主动展示购买链接。
- 用户主动找类似款时，展示多个候选链接。

### 4.5 AI试穿图生成服务

当前优先级：P0

能力：

- 调用 Image2 API。
- 根据用户参考图和穿搭方案生成效果图。
- 返回图片 URL。
- 记录生成状态。
- 记录失败原因。

第一阶段可以先支持：

- 一张主图生成。
- 三种场景切换生成。
- 免费 / 付费额度控制。

### 4.6 偏好学习服务

当前优先级：P1

能力：

- 记录评分。
- 记录收藏。
- 记录场景切换。
- 记录主题切换。
- 更新 `User_Fashion_Profile`。

第一阶段不需要复杂模型，先做规则沉淀。

## 5. 第一阶段后台能力

### 5.1 用户管理

当前优先级：P1

- 查看用户资料。
- 查看生成次数。
- 查看会员状态。
- 查看图片授权状态。

### 5.2 趋势库管理

当前优先级：P1

- 上传趋势参考图。
- 给趋势图打标签。
- 设置场景、风格、颜色、身材策略标签。
- 设置是否可用于推荐。

### 5.3 商品候选库

当前优先级：P1

- 维护同类型商品候选。
- 每种类型支持多个链接。
- 标注来源平台。
- 标注价格区间。
- 标注关键标签。

注意：

商品候选库第一阶段只作为“用户主动找类似款”时的补充，不作为首页主动导购。

## 8. 生成额度规则

### 免费用户

- 每天最多生成 2 张试穿效果图。
- 失败不扣次数。
- 可查看基础推荐理由。
- 可评分。

### 付费用户

- 每天最多生成 10 张试穿效果图。
- 可使用更多场景。
- 可查看更完整的穿搭解释。
- 可保存更多历史方案。

### 额度扣减时机

只有当 Image2 返回成功图片并写入 `Outfit_Generation` 后，才扣减额度。

## 9. 第一阶段验收标准

### 产品验收

- 用户能完成登录。
- 用户能上传三视图。
- 用户能看到身材分析结果。
- 用户能选择场景。
- 用户能生成试穿图。
- 用户能评分。
- 免费和付费额度规则生效。
- 不主动展示购买链接。

### 设计验收

- 首页首屏以试穿图为核心。
- 顶部信息不重叠。
- 人物头部不被控件遮挡。
- Logo 不明显遮挡人物。
- 三个主题都可读。

### 技术验收

- 接口可 Mock。
- 图片上传流程可跑通。
- 生成状态可追踪。
- 失败可重试。
- 用户数据和生成记录可保存。

## 10. 建议开发顺序

1. 固化首页 UI 和主题切换。
2. 建立用户与额度 Mock 数据。
3. 做三视图上传页。
4. 做身材分析接口 Mock。
5. 做推荐编排 Mock。
6. 接入或模拟 Image2 生成。
7. 做结果页和评分。
8. 沉淀偏好画像。
9. 再补衣橱和类似款入口。

## 11. 第一阶段完成定义

第一阶段完成不是“所有功能都很完整”，而是用户可以完成一次真实体验：

```text
我上传照片
→ 米粒知道我的比例
→ 我选择今天要去哪里
→ 米粒给我生成一张我穿上后的效果图
→ 我能评价它准不准、喜不喜欢
→ 第二天它能根据我的反馈再变好一点
```
