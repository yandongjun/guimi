# 首页导航与高级试衣棚设计规格

## 背景

MVP 的核心验证目标是：用户是否愿意持续打开小程序看每日穿搭。当前底部有 5 个 tab：今日、建档、点评、衣橱、我的。入口完整，但对 MVP 来说偏工具化，会分散“每日打开”的主路径。

首页原始海报风格成立，但 image2 生成后的 PNG 可能把灰白棋盘格画进图片内容里，而不是真 alpha 透明背景。仅靠提示词不能保证稳定，因此首页需要同时做两层处理：

- 提示词层：减少模型生成棋盘格、透明预览底、素材预览底的概率。
- 前端承接层：即使生成图带浅棋盘格，也用设计过的摄影棚背景、柔光和遮罩把画面接住。

## 目标

1. 将底部导航收敛为 4 个 tab：今日、衣橱、记录、我的。
2. 保留首页原有高级海报气质，不重做成普通信息流。
3. 首页生成图区域采用“高级试衣棚”视觉：暖白摄影棚光、柔和椭圆底、轻遮罩。
4. 生成图即使不是严格透明 PNG，也不能显得像素材预览或调试画面。
5. 修订图片生成 prompt，明确禁止透明棋盘格、灰白格背景、素材预览底。

## 非目标

- 不修改三视图上传对公网文件服务器的依赖。
- 不关闭或脱敏当前调试日志。
- 不新增完整衣橱管理、社区、电商推荐、连续风格计划。
- 不重构所有页面视觉系统。
- 不解决 image2 的真实透明能力问题；本规格只降低坏结果概率，并提升前端容错。

## 底部导航设计

### 最终 tab

1. 今日
   - 默认首页。
   - 承载每日穿搭、场景切换、重新生成、进入详情。

2. 衣橱
   - 承载用户已上传或 mock 的轻量衣橱。
   - 强化“推荐用了我的衣服”的产品心智。

3. 记录
   - 替代原“点评”的独立 tab。
   - 承载历史生成、反馈记录、穿搭评分回看。
   - MVP 第一版可先复用或轻量化 daily/evaluate 能力，后续再完善历史列表。

4. 我的
   - 承载用户资料、Plus、测试用户切换。
   - “建档”迁入我的页，作为高优先级任务卡展示。

### 页面迁移

- `pages/capture/capture` 保留页面，但不再作为 tab。入口放到“我的”和首页缺档提示中。
- `pages/evaluate/evaluate` 保留页面，但不再作为 tab。入口放到“记录”或穿搭详情页。
- `pages/profile/profile` 继续作为“我的”。
- 需要新增或改造 `pages/record/record`，用于替代 tab 中的“点评”。

## 首页高级试衣棚设计

### 视觉原则

首页保留现在的海报结构：左侧大标题、幸运色/天气胶囊、理由拆解、底部 MILI 品牌字样。生成图区域改为“试衣棚舞台”，不要直接依赖 PNG 透明效果。

### 画面结构

1. 背景层
   - 继续使用暖米色高级纸感底。
   - 人物所在区域增加暖白摄影棚光斑。
   - 人物脚下增加柔和椭圆地台/阴影，给人物落点。

2. 图片层
   - 生成图仍使用 `<image class="model">`。
   - 图片后方增加 `model-halo`，覆盖浅棋盘格边缘。
   - 图片上方增加轻遮罩 `model-veil`，用低透明度暖色渐变压掉灰白格噪声。

3. 文案层
   - 左侧标题缩短最大宽度，避免压到人物脸和上半身。
   - 衣橱来源文案保留，但改为更像杂志注释：例如“衣橱引用：奶油白西装 / 深靛牛仔裤”。
   - 右侧身体标注减少到 1-2 个，避免和生成图主体冲突。

4. 品牌层
   - `MILI.` 保留，但降低对人物脚部的遮挡风险。
   - 必要时根据生成图状态把品牌字样透明度降到更稳的范围。

### 生成前后状态

- 生成前：继续展示现有高级海报感，使用 mock/历史图。
- 生成中：锁住首页交互，保留上一张可用图，不切换到空白或棋盘图。
- 生成成功：加载新图，但舞台背景和遮罩始终存在。
- 生成失败：保留上一张图，toast 说明失败原因。

## Prompt 修订

在 `buildImagePrompt` 中保留现有人物身份、三视图一致、衣橱引用、场景和天气要求，新增明确约束：

- 不要生成透明棋盘格。
- 不要灰白格背景。
- 不要素材预览底。
- 如果无法输出真实 alpha 透明 PNG，则使用干净暖白纯色背景。
- 画面不要带截图 UI、水印、边框、网格、地面线或软件预览痕迹。

推荐改写方向：

```text
请输出真实透明背景 PNG；如果模型无法保证真实 alpha 透明，请改用干净暖白纯色摄影棚背景。禁止出现灰白棋盘格、透明预览格、素材软件预览底、截图 UI、水印、边框或网格。
```

## 数据与状态

本设计不新增后端数据结构。复用现有字段：

- `outfit.tryOnImage`
- `outfit.imageJob.status`
- `outfit.closetUsageCopy`
- `outfit.usedClosetItemLabels`
- `quota`
- `generationLocked`
- `generationHint`

首页可增加前端派生状态：

- `hasGeneratedImage`：是否已有真实生成任务返回图。
- `modelImageLoaded`：当前图片是否加载成功。
- `modelImageError`：当前图片是否加载失败。

这些状态只服务视觉与容错，不改变推荐逻辑。

## 错误处理

1. 图片加载失败
   - 保留上一张可用图。
   - 显示轻量 toast。
   - 不让首页进入空白。

2. 生成任务 pending
   - 页面按钮锁住。
   - 展示“搭配图生成中，请等待完成”。
   - 不扣配额，直到任务 ready。

3. 生成图包含棋盘格
   - 前端无法可靠识别所有棋盘格。
   - 通过舞台背景、柔光、遮罩降低视觉破坏。
   - 后续若需要，可增加服务端图像检测与二次去底，但不属于本次 MVP 范围。

## 测试标准

1. `app.json` tabBar 只保留 4 个 tab。
2. 首页缺档时仍能进入建档页。
3. 我的页能进入建档页。
4. 原点评能力不丢失，可从记录或详情进入。
5. image2 prompt 中包含禁止棋盘格和 fallback 暖白背景的约束。
6. 首页使用生成图时，视觉上看不到裸露的大面积棋盘格。
7. 生成中、失败、成功三个状态不出现布局跳动或空白首页。
8. 现有 Node 语法检查通过。

## 实施顺序

1. 修改 tabBar：从 5 tab 收敛到 4 tab。
2. 添加或改造记录页入口，迁移点评入口。
3. 在我的页增加建档任务入口。
4. 修订图片生成 prompt。
5. 优化首页 WXML/WXSS：加入高级试衣棚背景、光晕、遮罩、地台。
6. 验证生成前、生成中、生成失败、生成成功四种状态。
