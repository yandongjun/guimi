# 闺蜜 MVP 技术方案

## 当前结构

- 小程序前端：`pages/*`
- 前端服务层：`services/api.js`
- 本地 Mock 服务：`services/mock-server.js`
- Mock 数据：`data/mock.js`
- Node Mock 后台：`backend/server.js`
- 产品与接口文档：`docs/*`

小程序默认使用 `app.js` 里的 `useMock: true`，所有页面直接走本地 Mock 服务。要切换到后台接口，把 `useMock` 改成 `false`，并保证 `apiBaseUrl` 指向后端服务。

## 真实服务拆分

1. 用户与偏好服务
   保存登录用户、城市、星座、偏好色、风格关键词、尺码和隐私授权。

2. 身材档案服务
   接收三视图照片，调用视觉模型输出比例点、体态判断、身材标签和穿搭策略。

3. 衣橱服务
   管理单品、图片、颜色、品类、保暖度、正式度、适用场景和使用频率。

4. 穿搭推荐服务
   结合天气、身材、衣橱、趋势商品和偏好，产出分层搭配、原因和提醒。

5. 穿搭点评服务
   对用户穿搭照进行结构化评价，并把可识别单品写入候选衣橱。

6. 试穿图生成服务
   将用户三视图、单品图、商品图和推荐方案整理成 Image2 请求，生成效果图。

## AI 编排建议

核心链路不建议让一个大模型一次完成。更稳的方式是：

1. 视觉模型提取事实：比例、姿态、衣物、颜色。
2. 规则层做硬约束：天气、保暖、场景、已有衣物优先。
3. 审美模型做搭配选择：色彩、版型、风格一致性。
4. 文案模型解释原因：用闺蜜口吻输出可执行建议。
5. 生成模型出图：只负责把已确定的穿搭视觉化。

## 隐私与安全

- 三视图和穿搭照都属于敏感图片，正式版必须显式授权。
- 上传后应做对象存储私有化、短期签名访问和删除入口。
- 分析报告应避免羞辱性词汇，用“比例策略”替代“缺陷描述”。
- 商品推荐必须标记来源，避免把广告包装成客观审美建议。
